# Services Index Copy Deck

Use this file when rewriting or restructuring `/services` so it matches the new service-page system.

## Page job
The services index should:
- explain the main problem clusters
- help visitors recognize their situation
- route users to the right child-service page
- connect each cluster to proof
- move qualified users to `/contact?intent=service`

It should **not**:
- read like a flat capability list
- explain every service in equal depth
- repeat the homepage hero story
- lead with internal labels before the problem is clear

## Recommended page structure
1. Services hero / intro
2. Four problem clusters
3. Cluster-level proof anchors
4. Child-page route blocks
5. Final CTA

## 1. Services hero / intro

### Headline options
1. Services built around the bottlenecks that actually slow growth down.
2. The work usually starts in one of four places.
3. Strategy, websites, systems, and demand — connected instead of siloed.

### Subhead options
1. Most businesses do not need more activity. They need the right layer fixed first — whether that is strategy, website performance, CRM structure, or growth systems.
2. The visible problem is not always the real one. Weak conversion, unclear messaging, manual follow-up, and poor visibility tend to compound each other.

### Intro guidance
Keep the intro focused on:
- why services are grouped into clusters
- how the work connects across layers
- why buyers should choose the cluster that matches their bottleneck

## 3. Cluster card / section structure
For each cluster, include:
- small cluster label
- one clear problem-led heading
- 1–2 sentence explanation
- 2–4 child-page links
- one relevant proof link
- one CTA to `/contact?intent=service`

## 4. Final CTA

### Headline options
1. If you already know where the bottleneck is, let’s talk.
2. If you can name the problem, the next step is probably a conversation.
3. If one of these clusters already sounds familiar, we should talk through it.

### Support line options
- No pitch. Just a practical conversation about what is actually getting in the way.
- I reply within 1 business day, usually faster.

### CTA route
- `/contact?intent=service`

## Writing rules
- problem first, service name second
- one cluster = one distinct job
- avoid long lists of deliverables at the index level
- keep technical language in child pages, not high on the index
- attach one relevant proof example per cluster when possible

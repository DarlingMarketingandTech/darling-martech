# Service Page Template

Use this structure for every new service or child-service page.

## 1. Page role
- **Primary job:** explain the service clearly and convert the right buyer into a consultation request
- **Primary CTA:** `/contact?intent=service`
- **Secondary CTA:** related proof on `/work` or a tool on `/tools` only if naturally useful

## 2. Hero
- **H1:** plain-English service name
- **Subhead:** one sentence that explains what the service fixes
- **Micro-proof:** one relevant stat or short credibility line
- **CTA:** service-intent contact CTA

## 3. What this is
### Layer 1 — plain-English
Define the service in language a founder or operator can understand without jargon.

### Layer 2 — translator
Explain how it connects strategy, systems, execution, and growth.

### Layer 3 — technical
Explain the deeper implementation detail advanced buyers expect.

## 4. Why it matters
Focus on business impact, not just deliverables.
Examples:
- lost leads
- weak conversion
- wasted spend
- poor visibility
- manual work
- disconnected tools
- unclear reporting

## 5. Signs you need this
Use practical symptom bullets.
Examples:
- the website gets traffic but not enough inquiries
- the team is following up manually
- the CRM exists but no one trusts it
- marketing and operations are not connected

## 6. What this usually includes
Use bullets, not vague paragraphs.
Typical structure:
- audit / discovery
- strategy / design
- implementation / cleanup
- visibility / reporting / training / handoff

## 7. Technical scope
This is the place for specialist language.
Examples:
- CRM architecture
- lifecycle automation
- workflow orchestration
- attribution cleanup
- page architecture
- dashboard design
- structured content / schema / internal linking

## 8. Related proof
Include:
- 1 primary work proof item
- 1–2 supporting work proof items
- one sentence explaining why they are relevant to this service

## 9. FAQ
Use 4–6 questions that answer objections and capture search intent.
Examples:
- what does this include?
- how do I know if I need this vs a redesign?
- do you implement it or just advise?
- can this work with my current stack?
- how long does this usually take?

## 10. CTA block
Keep it simple:
- one line of reassurance
- one primary button to `/contact?intent=service`
- optional secondary proof link

## Writing rules
- business problem first
- translation before jargon
- specifics over buzzwords
- proof over hype
- one clear CTA
- avoid sounding like a generic agency page

## Minimum viable content for launch
- H1
- subhead
- one plain-English intro paragraph
- 4–6 signs-you-need-it bullets
- 5–7 what-it-includes bullets
- one technical scope section
- one proof block
- 4 FAQ items
- one CTA block

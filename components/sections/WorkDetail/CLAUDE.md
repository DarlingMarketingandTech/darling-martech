## Work Detail (`/work/[slug]`) — Local Guidance

**Role:** a proof page that explains what changed (problem → intervention → outcome) and then routes out cleanly. It should feel editorial, not like a case-study CMS.

### Template weights (choose the smallest effective template)
- **Flagship longform**: primary anchor cases. More narrative, more context, still proof-first.
- **Supporting standard**: normal case studies. Tight structure; no extra sections “because we can.”
- **System compact**: system-children inside a parent engagement. Lead with system role + outcome, keep scope tight.
- **System expanded**: only when the system-child needs more mechanism clarity (still avoid sprawling “feature tours”).

### Parent/child proof logic
- System-children should clearly show they are **inside** a parent engagement (breadcrumb/badge/backlink).
- System-children must not read like separate flagship stories; they inherit trust from the parent and add mechanism proof.

### Route-out discipline
- End with a **single route-out set**:
  - one service backlink when available
  - up to 2 related proof links (keep it curated)
- Avoid “related sprawl” (prev/next + grids + multiple rails). The page should close.

### Service backlink rules
- If the work page supports a service page, show exactly one “This build supports …” link.
- The backlink should resolve to the canonical service route (respect nested service routes).

### Pike Medical / Graston-style parent-system handling
- Treat **Pike Medical Consultants** and **Graston Technique** as system-parent archetypes:
  - parent page is the flagship story
  - children are system proof slices (compact/expanded) that route back to the parent context
- Do not let child pages compete with the parent’s proof signal or narrative.

### Anti-patterns to avoid
- Turning system-child pages into mini-flagships (too much narrative, too many sections, too many links)
- Adding multiple competing navigation systems (prev/next + “more systems” grids + “related” rails)
- Feature-tour copy that reads like product marketing instead of proof
- Breaking the “problem → intervention → outcome” flow by leading with tools/tech stack

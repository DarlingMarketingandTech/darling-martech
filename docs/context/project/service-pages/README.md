# Service Page Briefs

This folder is the execution-oriented source set for Darling MarTech service and sub-service content.

Use these files to:
- write or rewrite `app/services/[slug]` and future child-service pages
- keep service messaging problem-led instead of jargon-led
- connect each service to the right proof on `/work`
- keep page content layered for mixed literacy buyers

## Core writing model

Every page should move through 3 layers:

1. **Plain-English buyer language**
   - what is broken
   - what this fixes
   - why it matters
2. **Strategic translator language**
   - how the work connects strategy, systems, execution, and growth
   - what changes operationally after the work is done
3. **Technical / specialist language**
   - the implementation detail advanced buyers expect
   - architecture, automation, analytics, SEO, CRM, workflow, and tooling specifics

## Standard page structure

Each page should answer, in order:
1. What this is
2. Why it matters
3. Signs you need it
4. What it usually includes
5. Technical scope
6. Related proof
7. FAQs
8. CTA

See `service-page-template.md` for the reusable structure.

# Custom Tools / Apps Service Expansion Plan

Use this plan to evaluate whether Darling MarTech should add a service offer around custom tools, apps, and interactive products.

## Goal
Decide whether a new service should be added that captures:
- custom tools
- lightweight app/product builds
- interactive web experiences
- specialized digital products for businesses, creators, and teams

This is **not** a portfolio expansion plan.
It is a service-expansion plan.

## Core guardrail
Any project should be added only if it strengthens a service offer.
Do **not** add projects simply because they are interesting.
The site remains demand-generation-first, not portfolio-first.

## Recommended service framing
Preferred working label:
- **Custom Tools, Apps & Interactive Products**

Recommended positioning:
- For teams, founders, and brands that need something more specialized than a standard website or off-the-shelf software.

## Why this service may fit
This service can support current positioning by showing:
- strategic problem-solving
- custom build capability
- product thinking
- interactive systems work
- specialized digital execution when a normal website is not enough

## Best fit in current service system
Primary recommendation:
- position this as a **specialty / next-tier service expansion**
- likely under or adjacent to the current **CRM, Automation & AI** cluster

Why:
- it fits systems and custom execution more naturally than a generic websites cluster
- it complements CRM / agentic / audit work
- it avoids reframing the entire site as a general dev studio

## Candidate proof projects
Likely strongest first proof candidates:
1. `piko-artist-website-v3`
2. `guitar-app`
3. `pro-studio-mixer` (only if mature enough)

Supporting / adjacent but not primary:
- `blazing-star-therapy` (better fit for website strategy / trust-led site proof)

## Questions the next planning pass should answer
1. Is this a sellable service now, or just a direction?
2. Who is the buyer for this offer?
3. What problem does it solve that existing services do not already cover?
4. Which 1–2 proof projects are strong enough to support it now?
5. Should it be a standalone service page, a specialty page, or a sub-service under systems?

## Anti-patterns
- turning the site into a portfolio of side projects
- using hobby/prototype work as proof without buyer-problem framing
- adding a generic “app development” service with no strategic differentiator
- introducing this service before its role in the current system is clear

## Suggested sequence
1. service-definition pass
2. proof-candidate evaluation
3. helper/proof mapping for the selected projects
4. service page or service-expansion implementation
5. only then add corresponding work entries if needed

# Work Detail Page Template

Use this structure for `/work/[slug]` pages.

## Page role
- **Primary job:** prove capability through a concrete business story
- **Primary CTA:** route to the most relevant service page when the pairing is clear
- **Secondary CTA:** one contact path or one relevant related proof link
- **Do not turn this page into:** a portfolio gallery, a process diary, or a dashboard walkthrough with no commercial framing

## Two valid templates
Not every work page should use the same storytelling weight.

### 1. Flagship longform
Use for:
- `graston-technique`
- `pike-medical-consultants`
- `317-bbq`
- `hoosier-boy-barbershop`
- `russell-painting`
- any future flagship with a broad enough business story

### 2. System-child / compact detail
Use for:
- child builds that sit inside a parent engagement
- supporting system surfaces
- narrower tools, dashboards, automations, or internal workflow builds

Do not inflate compact system proof into a fake enterprise case study.

---

# A. Flagship longform template

## 1. Hero
Include:
- client name
- plain-English problem statement
- one-sentence intervention summary
- 2–3 business outcomes or proof signals
- scope/services tags only if they help orientation
- one contextual “This build supports …” service link when available

Hero rule:
Lead with the business change, not the tech stack.

## 2. Why this mattered
Open with business context:
- what was stuck
- what was at risk
- why the previous path was not enough

This section should help a buyer recognize themselves before the page gets more specific.

## 3. What changed
Prefer three clean sections:
1. constraint or problem
2. strategic move / build
3. result / operating impact

You can rename them, but keep the logic.

## 4. Proof media
Use media to prove the story, not decorate it.

Best examples:
- website screens
- conversion paths
- dashboard or workflow screens
- operational surfaces
- service-page crops that show trust or hierarchy clearly

Avoid:
- logo-only hero media
- decorative image walls with no story role
- low-value before/after gimmicks

## 5. Results and operating impact
Tie outcomes back to the business problem.
Examples:
- qualified demand improved
- bookings or inquiries improved
- operational overhead dropped
- trust and conversion improved together
- visibility became easier to convert

Do not dump metrics without interpretation.

## 6. Validation layer
Use one proof voice where relevant:
- short testimonial
- quote
- supporting metric recap
- business-context confirmation

One good validation element is stronger than several weak ones.

## 7. Related route-out
End simply:
- one service page link
- up to 2 related work links
- one clear CTA path

Do not add a giant recommendation wall.

# B. System-child / compact detail template

## 1. Hero
Include:
- project name
- one-line explanation of what the build was
- parent engagement context
- one useful outcome or operational signal

## 2. Where it sat in the system
Explain:
- what larger engagement it belonged to
- who used it
- what bottleneck it addressed

## 3. What it changed
Use a short sequence:
- before
- build / mechanism
- after

This can be brief, but it should still explain why the build mattered.

## 4. Focused media
Use 1–3 strong supporting visuals max.
Bias toward UI, workflow, and real operating surfaces.

## 5. Route back to parent + service
Always make it easy to:
- view the parent work story
- understand which service line this supports

Compact child pages should deepen the parent proof, not compete with it.

---

# Writing rules
- business problem first
- explain what the build unlocked, not just what it included
- translate complexity before naming tools
- keep tone calm and commercially clear
- let proof do the persuasion

# Anti-patterns
- starting with stack or feature list
- long process theater that says little
- treating every sub-project like a flagship
- adding too many related links
- gallery-first layouts with thin commercial context
- forcing the same section order onto pages that clearly need the compact template

# Minimum viable content
## Flagship longform
- hero with business framing
- context/problem section
- what changed section
- proof media section
- results section
- one service link
- 1–2 related work links

## System-child / compact detail
- hero with parent context
- short system-role section
- what changed section
- focused media
- route back to parent + relevant service
